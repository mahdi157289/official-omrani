6. User Stories and Use Cases
6.1 User Stories
As a First-time Customer,
I want to place an order without creating an account,
so that I can complete my purchase as quickly as possible.

As a First-time Customer who just placed an order,
I want to receive an email to claim my automatically created account,
so that I can view my order status and save my details for next time.

As a Returning Customer,
I want to log into my profile to see my order history and update my personal info,
so that I can re-order easily and manage my data.

As a Customer,
I want to switch the website language between Arabic, French, and English via a clear button,
so that I can browse comfortably in my preferred language.

As the Store Admin (Owner),
I want to view an analytics dashboard with sales trends and popular products,
so that I can make informed business decisions.

As a Store Editor (Employee),
I want to update product images and mark items as "sold out",
so that the website is always accurate without needing the owner.

As any User,
I want to see a pleasant, branded loading screen when the site is fetching data,
so that I know the site is working and not broken.

6.2 Use Cases
Use Case ID: UC-01
Title: Guest Checkout with Automatic Account Creation
Actor: First-time Customer
Preconditions: Customer has items in their cart.
Main Flow:

Customer proceeds to checkout.

Customer fills the order form (name, email, phone, delivery address).

System validates data and places the order.

System checks if a user with that email exists.

If not, system creates a User record with the provided details and a null password.

System generates a secure account claim token and emails it to the customer.

Customer receives order confirmation and a separate email to set their password and claim their account.
Alternative Flows:

5a. If the email exists, the order is attached to the existing account.
Postconditions: An order is created. A new user account is created in an "unclaimed" state.

Use Case ID: UC-02
Title: Admin Views Real-Time Sales Dashboard
Actor: Store Admin
Preconditions: Admin is logged into the secure /admin panel.
Main Flow:

Admin navigates to the "Analytics" tab in the right-side navigation bar.

System displays a dashboard with key widgets: Sales chart (time-based), Top Products table, Order Status summary, and Customer Acquisition metrics.

Admin can adjust the date range (e.g., last 7 days, this month).

Data updates in real-time as new orders come in.
Postconditions: Admin gains insights into business performance.

9. Technical Specifications
9.1 Tech Stack
Frontend: Next.js 15 (App Router), React 19, TypeScript.

Styling: Tailwind CSS, shadcn/ui component library.

Internationalization: next-intl.

Backend/Fullstack: Next.js API Routes, Payload CMS (self-hosted).

Database & ORM: PostgreSQL (on Render), Prisma.

Authentication: NextAuth.js (Auth.js).

Media Storage & CDN: Cloudinary (with configured upload presets).

Deployment: Render.com (Web Service + PostgreSQL database).

Animation: Framer Motion for interactive micro-animations.

9.2 Architecture Overview
System: Monolithic Next.js application containing frontend, backend API, and admin CMS.

Data Flow:

User visits site (Vercel edge network).

Next.js renders page (SSR/SSG), fetching data via Prisma from PostgreSQL (Render).

User action (e.g., add to cart) hits Next.js API route.

API route uses Prisma to mutate data.

Admin publishes change in Payload CMS.

Payload hook calls Next.js revalidation endpoint.

Updated content is served on next request.

API Structure: RESTful conventions within Next.js App Router API routes.

// ========== ENUMS ==========
enum UserRole {
  CUSTOMER
  EDITOR
  ADMIN
}

enum UserStatus {
  UNCLAIMED  // Account created via guest checkout, not yet activated
  ACTIVE
  INACTIVE
}

enum OrderStatus {
  PENDING
  CONFIRMED
  PREPARING
  READY_FOR_PICKUP
  OUT_FOR_DELIVERY
  DELIVERED
  CANCELLED
}

enum DeliveryType {
  PICKUP_KAIROUAN
  PICKUP_TUNIS
  DELIVERY
}

// ========== MAIN TABLES ==========

model User {
  id            String      @id @default(cuid())
  email         String      @unique
  name          String?
  phone         String?
  password      String?     // Nullable for UNCLAIMED users
  role          UserRole    @default(CUSTOMER)
  status        UserStatus  @default(ACTIVE)
  addresses     Address[]
  orders        Order[]
  createdAt     DateTime    @default(now())
  updatedAt     DateTime    @updatedAt

  @@map("users")
}

model Address {
  id          String   @id @default(cuid())
  userId      String
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  label       String?  // e.g., "Home", "Work"
  fullAddress String
  city        String
  postalCode  String?
  isDefault   Boolean  @default(false)
  createdAt   DateTime @default(now())

  @@map("addresses")
}

model Product {
  id            String   @id @default(cuid())
  sku           String   @unique
  nameFr        String
  nameAr        String
  slug          String   @unique
  descriptionFr String   @db.Text
  descriptionAr String   @db.Text

  isFeatured    Boolean  @default(false)
  categoryId    String
  category      Category @relation(fields: [categoryId], references: [id])
  images        Media[]
  orderItems    OrderItem[]
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

}

model Category {
  id            String    @id @default(cuid())
  nameFr        String
  nameAr        String
  slug          String    @unique
  descriptionFr String?   @db.Text
  descriptionAr String?   @db.Text
  isActive      Boolean   @default(true)
  products      Product[]
  createdAt     DateTime  @default(now())

  @@map("categories")
}

  altTextAr    String?

  product      Product? @relation(fields: [productId], references: [id], onDelete: Cascade)
  eventId      String?

  id            String   @id @default(cuid())
  titleFr       String
  titleAr       String

  startDate     DateTime
  endDate       DateTime?
  descriptionFr String   @db.Text
  descriptionAr String   @db.Text
  isActive      Boolean  @default(true)
  imageId       String?
  image         Media?   @relation(fields: [imageId], references: [id])
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

model Order {
  id              String      @id @default(cuid())
  orderNumber     String      @unique // Format: MO-YYYY-MM-XXXXX
  userId          String?     // Nullable for pure guest orders (optional link)
  user            User?       @relation(fields: [userId], references: [id])
  customerName    String
  customerEmail   String
  customerPhone   String
  deliveryType    DeliveryType
  deliveryAddress String?     // Full address string for delivery
  deliveryZone    String?     // For fee calculation

  deliveryFee     Decimal
  totalAmount     Decimal

  id            String  @id @default(cuid())

9.2 Detailed System Architecture
The platform follows a modern, monolithic full-stack architecture using Next.js, optimized for performance and developer experience.

Data Flow Explanation:

Static & Dynamic Rendering:

Public pages (Home, Story, Events) are Statically Generated (SSG) at build time for instant loads.

Product listings and detail pages use Incremental Static Regeneration (ISR) or Server-Side Rendering (SSR) to stay fresh.

User Interactions:

Browsing: Fetches data from the database via Prisma in API routes or server components.

Adding to Cart: Hits /api/cart/add, which manages state in the user's session or localStorage.

Checkout: /api/checkout/session calculates fees. /api/checkout/order creates the Order and OrderItem records, and potentially a User.

Admin Operations:

The Payload CMS Admin (/admin) provides a direct UI to manage all database tables.

When an admin publishes a change, a Payload afterChange hook calls the secured /api/admin/revalidate endpoint.

This triggers Next.js to revalidate the affected pages, making updates live instantly.

Media Flow:

Admins upload images within Payload CMS.

Payload uses the Cloudinary upload preset to send files directly to Cloudinary.

Cloudinary returns an optimized URL, which Payload saves in the Media table.

The frontend displays images via Cloudinary's CDN with automatic format/quality optimization.

Authentication Layer:

NextAuth.js wraps admin API routes and pages. Users logging into /admin are validated against the User table (with role: ADMIN/EDITOR).

Customer profile routes (/api/user/*) are also protected, allowing users to manage their own data.
